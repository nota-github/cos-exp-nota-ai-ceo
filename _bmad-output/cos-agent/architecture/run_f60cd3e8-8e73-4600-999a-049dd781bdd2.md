# Nota AI CEO용 전략 인텔리전스 시스템 Architecture v0

## Document Purpose
이 문서는 승인된 brief와 최신 PRD를 바탕으로 `Nota AI CEO용 전략 인텔리전스 시스템`의 구현 지향 아키텍처 제약을 정의한다.  
목표는 후속 구현 단계가 흔들리지 않도록 시스템 경계, 데이터 흐름, 저장소 책임, 런타임 제약, 실패 처리 원칙을 먼저 고정하는 것이다.

입력 기준:
- Latest brief: `run_4ffd1ecb-460a-434a-93e0-48c1487803ef.md`
- Latest PRD: `run_20e3fe9c-4741-4902-b156-8b8e1d46f5d9.md`

현재 워크스페이스는 README 외 제품 코드가 거의 없는 그린필드 상태이므로, 본 문서는 특정 기존 구현을 설명하기보다 `무엇을 어떤 경계로 만들 것인가`를 명확히 하는 데 집중한다.

## Architecture Overview

### System Goal
이 시스템은 일반 뉴스 수집기가 아니라, 한국 영업일 기준으로 AI/반도체 및 Nota AI 관련 변화만 선별하고, 다중 출처로 검증한 뒤, Nota AI 관점의 전략적 해석과 대응 권고를 대시보드와 문서형 리포트로 동시에 발행하는 `daily strategic intelligence pipeline`이어야 한다.

### Recommended Architectural Style
초기 버전은 `배치 중심 파이프라인 + 공통 canonical briefing snapshot + 읽기 전용 전달 채널` 구조를 채택한다.

핵심 이유는 다음과 같다.
- PRD의 핵심은 실시간 스트리밍보다 `매일 누락 없이, 설명 가능하게, 비교 가능한 브리핑`이다.
- 대시보드와 문서형 리포트가 동일한 메시지를 공유해야 하므로, 채널별 별도 생성 로직보다 `하나의 일일 스냅샷`이 더 안전하다.
- 사람의 사전 검토 없이 자동 발행해야 하므로, 즉흥적 UI 생성보다 `검증된 중간 표현(canonical model)`을 두는 편이 오류를 통제하기 쉽다.
- 이슈의 지속/심화/완화 추적이 필요하므로, 일회성 요약보다 `이슈 레지스트리`와 `브리핑 버전`이 중심이 되어야 한다.

### Core Architecture Decisions
1. `배치 우선`: 실시간 알림 시스템이 아니라, 한국 영업일 기준의 일일 생성 파이프라인으로 설계한다.
2. `하나의 소스 오브 트루스`: 대시보드와 문서형 리포트는 동일한 `briefing_snapshot`에서 파생되어야 한다.
3. `검증 후 해석`: 원시 수집 후 바로 요약하지 않고, 먼저 정규화, 관련성 평가, 이슈 클러스터링, 다중 출처 검증을 통과시킨 후 전략 해석을 수행한다.
4. `Fail-closed major issue policy`: 핵심 이슈는 다중 출처 검증을 통과하지 못하면 상위 섹션에 노출하지 않고, 워치리스트나 유보 상태로 남긴다.
5. `Greenfield pragmatism`: 초기 버전은 과도한 분산 시스템 대신 단일 코드베이스 내 API/worker 분리와 관계형 저장소 중심으로 설계한다.

### Logical Flow
```text
KST Business-Day Scheduler
  -> Source Connectors / Ingestion
  -> Normalization + Taxonomy Tagging
  -> Nota AI Relevance / Impact Scoring
  -> Issue Clustering + Cross-Source Validation
  -> Strategic Interpretation + Recommendation Generation
  -> Canonical Briefing Snapshot Assembly
  -> Publish to Dashboard View + Document Report
  -> Persist History, Evidence, Run Logs
```

### Architecture Principles
- 모든 사용자 채널은 같은 날짜 범위와 같은 우선순위를 공유해야 한다.
- 모든 상위 이슈는 `무엇이 변했는가`, `왜 중요한가`, `Nota AI 어떤 영역에 영향이 있는가`, `권고안`, `근거 출처`, `불확실성`을 구조적으로 담아야 한다.
- 시스템은 `새로움`보다 `정확성과 추적성`을 우선한다.
- 주요 의사결정은 LLM 단독 추론이 아니라 규칙, 메타데이터, 검증된 근거 집합을 바탕으로 생성되어야 한다.
- 자동 발행 전제이므로, 생성 실패보다 `불충분 근거를 명시한 축소 발행`이 더 바람직하다.

## Major Components

### 1. Strategy Context Registry
Nota AI 중심 해석을 가능하게 하는 운영 설정 레이어다. 일반 뉴스 요약기와 가장 크게 다른 부분이다.

책임:
- Nota AI 사업 축, 제품 축, 파트너십 축, 투자자 커뮤니케이션 축, 경쟁 포지셔닝 축을 정의한다.
- 핵심 경쟁사, 파트너, 기술 키워드, 공급망 키워드, 금지/저우선 키워드를 관리한다.
- AI/반도체 직접 관련 정책만 포함하도록 범위 규칙을 저장한다.
- relevance scoring과 recommendation prompt의 기준 컨텍스트를 제공한다.

구현 제약:
- 코드 하드코딩이 아니라 운영 조정 가능한 설정 저장소여야 한다.
- 설정 변경은 버전 관리되어야 하며, 각 브리핑이 어떤 설정 버전을 사용했는지 추적 가능해야 한다.

### 2. Calendar and Run Orchestrator
한국 영업일 기준의 일일 산출을 보장하는 실행 제어 레이어다.

책임:
- 대한민국 공휴일 및 주말을 고려한 영업일 캘린더를 관리한다.
- `브리핑 대상 기간(covered_from, covered_to)`을 계산한다.
- 일일 자동 실행, 수동 재실행, 실패 후 재시도 정책을 관리한다.
- 동일 영업일에 대해 중복 발행이 생기지 않도록 idempotency를 보장한다.

구현 제약:
- 공휴일 브리핑을 건너뛰되, 다음 영업일 브리핑에 누적 반영해야 한다.
- 발행 시각은 운영 설정으로 변경 가능해야 하지만, 기준 타임존은 KST로 고정한다.
- 각 run은 `planned`, `running`, `completed`, `degraded`, `failed`, `published` 상태를 가져야 한다.

### 3. Source Connectors and Ingestion Layer
외부 신호를 수집하는 입력 계층이다.

대상 소스 범주:
- 공식 기업 발표, IR, 보도자료, 블로그
- 주요 뉴스/와이어
- 반도체/AI 산업 전문 매체
- 규제/정부 기관의 산업 직접 영향 공지
- 시장/자본시장 시그널용 신뢰 가능한 데이터 피드

책임:
- 소스별 수집 어댑터를 제공한다.
- 원문 메타데이터와 수집 시각을 저장한다.
- 중복 기사/재배포 기사/단순 요약 전재를 식별하기 위한 source family 정보를 부여한다.

구현 제약:
- 원문 전문 전체를 장기 저장하지 못하는 라이선스 상황을 고려해, 본문 전문이 아닌 메타데이터, 요약 추출, 스냅샷 참조를 기본으로 한다.
- 동일 내용을 재배포한 기사 여러 건을 `독립 출처`로 오인하면 안 되므로, URL 개수보다 `source_family` 단위 검증이 필요하다.

### 4. Normalization and Taxonomy Layer
수집된 원시 항목을 후속 판단이 가능한 공통 구조로 바꾸는 계층이다.

책임:
- 제목, 발행시각, 발행처, 원문 링크, 언어, 엔터티, 산업 태그를 표준화한다.
- AI/반도체/경쟁/시장/정책 범주 태그를 부여한다.
- 노이즈 제거, 중복 병합 후보 생성, 언어 차이 정규화를 수행한다.

구현 제약:
- LLM 호출 전에 규칙 기반 정규화와 태깅을 먼저 수행해 비용과 변동성을 낮춘다.
- `일반 정책`과 `산업 직접 영향 정책`을 구분할 수 있는 분류 규칙이 필요하다.

### 5. Relevance and Prioritization Engine
`이 정보가 Nota AI에 왜 중요한가`를 점수화하는 계층이다.

평가 축:
- 사업 기회 또는 리스크
- 제품/기술 전략 영향
- 파트너십/공급망 영향
- 투자자 커뮤니케이션 영향
- 경쟁 포지셔닝 영향

책임:
- 각 normalized signal에 Nota AI 관련성 점수를 부여한다.
- 상위 이슈 후보와 저우선 관찰 항목을 구분한다.
- 고객 산업/거시경제/글로벌 자본시장 신호는 직접 연관성이 명확할 때만 상위 후보로 올린다.

구현 제약:
- 완전한 LLM 블랙박스 점수만 쓰지 않고, 규칙 기반 feature와 LLM 판단을 결합한 hybrid scoring을 사용한다.
- 점수 결과만 저장하지 말고, 왜 해당 점수가 나왔는지 설명 가능한 이유 필드를 함께 저장해야 한다.

### 6. Issue Clustering and Cross-Source Validation Engine
여러 신호를 하나의 `이슈`로 묶고, 핵심 이슈로 올릴 수 있는지 판단하는 계층이다.

책임:
- 동일 사건을 다룬 여러 항목을 하나의 issue candidate로 군집화한다.
- 검증 상태를 `unverified`, `watchlist`, `validated_major` 등으로 구분한다.
- 이전 영업일의 issue registry와 비교해 `new`, `continuing`, `escalating`, `easing` 상태를 판정한다.

핵심 규칙:
- `major issue`는 최소 2개 이상의 독립 source family 근거가 있어야 한다.
- 가능한 경우 최소 1개의 1차/공식 출처를 포함하는 것이 바람직하다.
- 검증 부족 시 상위 확정 이슈가 아니라 워치리스트에 남긴다.

구현 제약:
- `출처 수`와 `출처 독립성`은 다르므로, validation rule은 source family와 근거 유형을 함께 본다.
- 이슈 군집 결과는 다음 영업일 비교를 위해 안정적인 cluster key 또는 issue fingerprint를 가져야 한다.

### 7. Strategic Interpretation and Recommendation Engine
검증된 이슈를 CEO용 전략 메모로 변환하는 계층이다.

책임:
- 각 validated issue에 대해 사실 요약, 중요성 해석, Nota AI 영향, 권고안, 리스크/주의사항, 확신도를 생성한다.
- 권고안은 `유지`, `대응`, `실험`, `보류`와 같은 의사결정 가능한 형태로 정규화한다.
- 공격적 기회 탐색 관점을 유지하되, 안정성 리스크를 함께 서술한다.

구현 제약:
- 프롬프트는 issue evidence set과 Nota AI context registry를 입력으로 사용해야 한다.
- 생성 결과는 자유 텍스트만 저장하지 말고 구조화된 필드로 파싱해야 한다.
- 증거가 부족한 주장, 과도한 단정, 출처 없는 수치 인용은 schema validation에서 차단해야 한다.

### 8. Brief Composer and Publisher
검증 및 해석된 이슈를 하루치 산출물로 묶는 계층이다.

책임:
- `executive summary`, `major issues`, `watchlist`, `recommendation summary`, `coverage window`, `method note`를 구성한다.
- 대시보드와 문서형 리포트가 공유할 canonical briefing snapshot을 생성한다.
- 문서형 리포트용 Markdown/HTML 아티팩트와 대시보드용 JSON/read model을 동시에 발행한다.

구현 제약:
- 채널별 별도 카피 생성 금지. 채널 표현 차이는 허용하되 의미 차이는 허용하지 않는다.
- 한 영업일 브리핑은 불변 snapshot으로 저장하고, 재발행 시 새 revision을 만든다.

### 9. Delivery Surfaces
초기 전달 채널은 두 개다.

대시보드:
- 빠른 스캔, 이슈 드릴다운, 연속성 확인에 최적화한다.
- 사용자는 오늘 브리핑, 최근 영업일 브리핑, 이슈별 근거 링크를 볼 수 있어야 한다.

문서형 리포트:
- 공식 공유, 아카이빙, 경영 메모 용도에 최적화한다.
- canonical snapshot에서 직접 렌더링되어야 하며, 별도 편집본을 만들지 않는다.

구현 제약:
- 초기 버전의 대시보드는 읽기 전용이면 충분하다.
- 문서형 리포트의 canonical source는 Markdown으로 두고, 필요 시 HTML/PDF로 렌더링한다.

### 10. Observability and Operations Controls
자동 발행 시스템에서 품질을 운영적으로 통제하는 계층이다.

책임:
- run 성공/실패, source coverage, validation pass rate, LLM failure, publish success를 기록한다.
- 수집량 급감, 특정 소스 장애, 상위 이슈 0건, 리포트 생성 실패 같은 이상 상황을 탐지한다.
- 수동 rerun, draft inspection, degraded publish 판단을 지원한다.

구현 제약:
- 사람이 사전 승인하지 않더라도, 운영자는 왜 해당 브리핑이 그렇게 나왔는지 추적 가능해야 한다.
- 최소한 `run log`, `prompt/template version`, `source count`, `validation failure reason`은 남겨야 한다.

## Data, State, or Persistence Concerns

### Recommended Persistence Model
초기 버전의 시스템 오브 레코드는 PostgreSQL로 둔다.  
이유는 다음과 같다.
- 일일 배치, 구조화 데이터, 버전 추적, 관계 질의가 핵심이다.
- 이슈-근거-브리핑 간 연결을 강하게 유지해야 한다.
- 초기 범위에서는 별도 이벤트 버스나 검색 전용 저장소보다 운영 단순성이 더 중요하다.

보조 저장소:
- S3 호환 object storage: 원시 payload 스냅샷, 렌더링된 리포트 아티팩트, 디버그 산출물 저장

초기 버전에서 필수 아님:
- 별도 벡터 DB
- 별도 검색 엔진
- 복잡한 스트림 처리 인프라

### Core Domain Records
최소한 아래 수준의 데이터 모델이 필요하다.

| Entity | Purpose | Key Fields |
| --- | --- | --- |
| `source_catalog` | 수집 소스 정의 | source_id, source_type, source_family, trust_tier, active |
| `source_item` | 원시 수집 항목 | source_item_id, source_id, published_at, url, title, raw_ref, language |
| `normalized_signal` | 정규화된 신호 | signal_id, source_item_id, tags, entities, summary, dedupe_key |
| `relevance_assessment` | Nota AI 관련성 판단 | signal_id, relevance_score, impact_dimensions, reasoning |
| `issue_candidate` | 이슈 군집 중간 상태 | issue_id, fingerprint, title, status, trend_state, confidence |
| `issue_evidence` | 이슈별 근거 묶음 | issue_id, signal_id, evidence_role, source_family |
| `briefing_run` | 일일 실행 단위 | run_id, business_date, covered_from, covered_to, status, config_version |
| `briefing_snapshot` | 하루치 canonical 산출물 | snapshot_id, run_id, revision, executive_summary, publish_status |
| `briefing_issue` | 브리핑 내 노출 이슈 | snapshot_id, issue_id, priority_rank, section, recommendation_type |
| `report_artifact` | 문서형 리포트 결과물 | artifact_id, snapshot_id, format, storage_ref |
| `strategy_context_version` | Nota AI 관점 설정 | version_id, competitors, priority_themes, exclusions |

### Source Independence as First-Class Data
PRD의 다중 출처 교차검증 요구사항을 만족하려면, `URL 두 개`가 아니라 `독립 근거 두 개`를 판단할 수 있어야 한다.

따라서 저장 모델은 다음을 포함해야 한다.
- `source_family`: Reuters 재배포 기사 여러 건을 하나의 family로 취급 가능해야 한다.
- `evidence_role`: 공식 발표, 언론 보도, 분석 기사, 시장 데이터 등 근거 유형 구분
- `trust_tier`: 공식/1차/2차 근거 수준 구분

이 모델이 없으면, 자동 시스템이 겉보기 다수 기사에 속아 검증 완료로 오판할 위험이 크다.

### Immutability and Versioning
- 하나의 영업일 브리핑은 불변 snapshot으로 저장한다.
- 같은 영업일에 재실행해 내용이 바뀌면 revision을 올린다.
- 대시보드와 리포트는 항상 특정 snapshot revision에 연결되어야 한다.
- prompt template, scoring rule, strategy context version도 run 단위로 고정 저장한다.

### Historical Continuity
PRD의 `신규/지속/심화/완화` 요구를 만족하려면 브리핑 본문만 저장해서는 안 된다.

필수 상태:
- issue fingerprint 또는 stable cluster identifier
- 이전 브리핑 대비 변화 이유
- 이전 권고안과 현재 권고안의 차이
- 최근 N 영업일 이슈 출현 이력

### Failure and Degraded State Persistence
자동 발행 시스템은 `완벽 실패`와 `축소 발행`을 구분해야 한다.

예시:
- 일부 소스 수집 실패, 그러나 충분한 근거로 브리핑 생성 가능: `degraded`
- 핵심 검증 단계 실패로 major issue 생성 불가: 축소 브리핑 발행 또는 publish hold
- 저장소나 렌더링 실패: snapshot은 생성되었으나 artifact publish 실패

이 상태는 운영에서 후속 진단이 가능하도록 영속 저장되어야 한다.

## Integration and Runtime Constraints

### Recommended Technology Shape
초기 구현은 다음 형태를 권장한다.

- 언어: Python 3.12+
- API/웹 계층: FastAPI
- 서버 렌더링 대시보드/리포트 뷰: Jinja2 또는 유사 템플릿 엔진
- 배치/worker 계층: 동일 코드베이스의 별도 worker entrypoint
- DB: PostgreSQL
- Object storage: S3-compatible

이 선택의 이유:
- 수집, 정규화, 점수화, LLM orchestration, 문서 렌더링을 한 언어에서 다루기 쉽다.
- 초기 대시보드는 읽기 전용이므로, 별도 SPA/프론트엔드 앱 분리보다 서버 렌더링이 빠르고 단순하다.
- 후속 단계에서 richer UI가 필요하면 JSON read API를 유지한 채 분리할 수 있다.

### Deployment Boundary
초기 배포 단위는 최소 2개면 충분하다.
- `api`: 대시보드/리포트 조회, health, 운영 조회
- `worker`: 수집, 검증, 생성, 발행

공유 요소:
- domain model
- DB schema
- prompt/template registry
- strategy context registry

초기에는 마이크로서비스 분해를 하지 않는다.  
마이크로서비스는 소스 수, 실행 빈도, 팀 규모가 커질 때 다시 검토한다.

### Scheduling and Business Window Rules
- 기준 타임존은 `Asia/Seoul`로 고정한다.
- 브리핑 대상 기간은 `직전 성공한 영업일 cutoff 이후`부터 `금일 cutoff까지`로 계산한다.
- 공휴일/주말에는 발행하지 않고, 다음 영업일에 누적 기간을 한 번에 다룬다.
- 수동 rerun이 있어도 business date는 바뀌지 않아야 한다.

### External Integration Constraints
- 소스 연결은 connector abstraction 뒤에 감싼다. 특정 API나 RSS가 바뀌어도 상위 파이프라인은 유지되어야 한다.
- rate limit, robots, 라이선스 제약을 고려한 polite fetching이 필요하다.
- 본문 전문 보관이 불가한 소스는 요약/인용 최소 정보만 저장한다.
- 공식/1차 출처를 가능한 우선 수집 대상으로 둔다.

### LLM Runtime Constraints
LLM은 다음 역할로 제한한다.
- 정규화 보조
- 이슈 요약 및 전략 해석
- 권고안 표현 정리

LLM에 맡기지 말아야 할 것:
- 영업일 계산
- 출처 독립성 판정의 최종 결정
- 저장소 일관성 관리
- publish 여부의 최종 게이트

운영 제약:
- structured output schema 강제
- citation-required prompt design
- max tokens와 재시도 정책 분리
- prompt/template 버전 고정

### Channel Consistency Constraints
대시보드와 문서형 리포트가 달라질 수 있는 것은 `표현 형식`뿐이다.  
달라지면 안 되는 것은 아래와 같다.
- business date
- coverage window
- executive summary의 핵심 메시지
- major issues의 집합과 순서
- recommendation type과 confidence

따라서 UI/리포트는 raw query 조합으로 실시간 계산하지 말고, 발행된 snapshot만 읽어야 한다.

### Security and Access Assumptions
초기 버전의 핵심 입력은 대부분 외부 공개 정보다.  
그러나 Nota AI 관점 설정과 권고안은 내부 전략 맥락을 반영할 수 있으므로 아래 원칙을 둔다.
- `strategy_context_registry`는 일반 원시 뉴스 데이터와 구분해 접근 제어한다.
- 운영자가 아닌 일반 조회 사용자는 설정 원문보다 결과 요약만 보게 할 수 있다.
- 프롬프트/설정에 내부 민감 문구가 포함될 경우, raw prompt 보존 범위를 최소화한다.

## Implementation Risks and Tradeoffs

### 1. Batch-First vs Real-Time
선택:
- 초기 버전은 batch-first

장점:
- PRD의 일일 브리핑 품질, 비교 가능성, 검증 가능성에 맞다.
- 운영 복잡도가 낮고 품질 관리가 쉽다.

대가:
- 속보성은 희생된다.
- 장중 즉시 반응용 제품으로는 부적합하다.

판단:
- 현재 성공 기준은 `실시간 대응`이 아니라 `매일 전략 판단`이므로 타당한 절충이다.

### 2. Single Canonical Snapshot vs Channel-Specific Generation
선택:
- 공통 snapshot 채택

장점:
- 대시보드/리포트 불일치 위험을 줄인다.
- 이력 비교와 revision 관리가 단순하다.

대가:
- 채널별 개별 최적화 자유도가 줄어든다.

판단:
- 초기 CEO 전달 품질에서는 자유도보다 일관성이 더 중요하다.

### 3. PostgreSQL-Centric Storage vs Search/Vector-First
선택:
- Postgres 중심

장점:
- run, issue, evidence, revision 같은 관계형 추적성이 강하다.
- 초기 운영 복잡도를 낮춘다.

대가:
- 고급 의미 검색이나 대규모 문서 탐색에는 제한이 있다.

판단:
- v1의 주 업무는 탐색보다 `하루치 검증된 브리핑 생성`이다.

### 4. Server-Rendered Dashboard vs Separate Frontend App
선택:
- 초기에는 server-rendered dashboard

장점:
- 코드베이스, 배포, 인증 경계를 단순화한다.
- 보고서형 UX와 잘 맞는다.

대가:
- 복잡한 상호작용 분석 기능에는 한계가 있다.

판단:
- 현재 사용자는 CEO 1인 중심이며, 우선순위는 시각적 화려함이 아니라 빠른 전달이다.

### 5. Automated Direct Delivery vs Human Review
선택:
- 자동 직접 발행 유지

장점:
- 성공 기준과 일치한다.
- 운영 속도가 빠르다.

대가:
- 품질 게이트가 시스템 내부에 더 강하게 들어가야 한다.
- 잘못된 권고가 바로 노출될 수 있다.

완화:
- evidence threshold
- uncertainty/confidence 명시
- degraded publish 상태
- run audit trail

### 6. Aggressive Opportunity-Seeking Tone vs Risk Control
선택:
- 권고안은 기회 탐색 우선이되, 리스크 표기 의무화

장점:
- CEO용 전략 인텔리전스의 차별점을 살린다.

대가:
- 검증이 약한 신호를 과대평가할 유인이 생긴다.

완화:
- recommendation은 항상 `opportunity`, `risk`, `execution caveat` 3요소를 함께 저장한다.

## Constraints for Next Implementation Phase
다음 단계의 구현 스토리는 아래 순서를 깨지 않는 것이 중요하다.

1. 먼저 `strategy context`, `business calendar`, `briefing run`, `issue/evidence`, `briefing snapshot` 같은 핵심 도메인 모델을 고정해야 한다.
2. 그 다음 수집보다 `canonical snapshot을 어떤 규칙으로 생성/발행하는가`를 확정해야 한다.
3. 대시보드와 리포트는 별도 생성물이 아니라 `동일 snapshot의 두 표현`으로 잘라야 한다.
4. LLM 통합은 저장 모델과 validation rule이 선행된 뒤 붙여야 한다.
5. 운영상 가장 먼저 필요한 실패 처리는 `검증 부족 시 watchlist 강등`과 `degraded publish`다.

즉, 다음 단계의 구현은 `UI부터`가 아니라 `도메인 모델과 검증 파이프라인부터` 진행되어야 한다.

## Summary
이 시스템의 핵심은 많은 뉴스를 모으는 것이 아니라, `검증 가능한 이슈 단위의 전략 브리핑`을 매 영업일 반복 생산하는 것이다.  
이를 위해 초기 아키텍처는 다음 세 가지를 강하게 고정해야 한다.

- `공통 canonical briefing snapshot`
- `다중 출처 검증이 선행되는 issue registry`
- `KST 영업일 기준의 일일 batch pipeline`

이 세 축이 고정되면, 이후 스토리 단계에서 수집기, 검증 로직, 권고 생성, 대시보드, 리포트가 서로 충돌하지 않고 구현될 수 있다.
