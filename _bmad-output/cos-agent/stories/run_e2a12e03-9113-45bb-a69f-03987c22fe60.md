# Nota AI CEO용 전략 인텔리전스 시스템 Stories v0

## Overview
- Source basis: latest brief `run_4ffd1ecb-460a-434a-93e0-48c1487803ef.md`, latest PRD `run_20e3fe9c-4741-4902-b156-8b8e1d46f5d9.md`, latest architecture `run_f60cd3e8-8e73-4600-999a-049dd781bdd2.md`.
- Workspace state: current repository is effectively greenfield and contains only `README.md`, so the stories below include initial project scaffolding.
- Decomposition rule applied: UI, persistence, ingestion, validation, generation, publishing, and recovery work are split into separate stories unless they are inseparable for validation.
- Recommended package boundary for implementation: `src/nota_intel/` with shared domain modules, a FastAPI read surface, and a worker entrypoint in the same codebase.

## Epics

## Epic 1 - Platform Foundation and Run Control
Epic Key: epic-1
- Establish the Python/FastAPI/PostgreSQL skeleton, core execution persistence, strategy context versioning, KST business-day logic, and run lifecycle control.
- This epic must complete before ingestion or dashboard work starts because every later story depends on stable run, date, and revision semantics.

## Epic 2 - Source Intake and Normalization
Epic Key: epic-2
- Build the source catalog, connector contract, ingestion pipeline, normalization layer, and source-family dedupe needed for trustworthy downstream validation.
- This epic turns raw industry signals into normalized, de-duplicated records that later stories can score and cluster safely.

## Epic 3 - Issue Evaluation and Canonical Briefing
Epic Key: epic-3
- Score relevance to Nota AI, cluster related signals, validate issues across independent sources, classify trend state, and assemble the immutable daily snapshot.
- This epic creates the canonical intelligence object that both delivery channels must share.

## Epic 4 - Publishing and Executive Delivery Surfaces
Epic Key: epic-4
- Render the document report, publish shared snapshot revisions, expose read APIs, and build the CEO-facing dashboard and issue drilldown pages.
- This epic stays read-only and snapshot-driven so the dashboard and report cannot diverge semantically.

## Epic 5 - Operations and Delivery Safety
Epic Key: epic-5
- Add auditability, quality telemetry, degraded publish handling, and rerun controls needed for safe automated delivery without human pre-review.
- This epic hardens the MVP after the primary pipeline and delivery surfaces exist.

Story Key: story-5.2
Epic Key: epic-5
Required for Success: no
Depends On: story-1.5, story-4.2, story-5.1

Goal:
- Support safe degraded publication and explicit reruns for the same business date so the system can keep operating when some upstream checks fail.

In Scope:
- Implement degraded publish decision rules for cases such as partial source failure or zero validated major issues.
- Add a manual rerun command or endpoint that reuses the same business date and creates a new snapshot revision.
- Track whether the currently published result for a business date is normal or degraded.

Out of Scope:
- Human pre-approval workflows.
- Automatic alert routing to external channels.
- Self-healing retries for every stage of the pipeline.

Acceptance Criteria:
- A run with partial upstream failure can publish a degraded result only if validation rules for major issues are still enforced.
- Unvalidated issue candidates never appear as major issues just because the run is degraded.
- A manual rerun preserves the original business date and produces a new revision rather than overwriting the published one.
- Operators can tell from stored state whether the active published snapshot is normal or degraded.
